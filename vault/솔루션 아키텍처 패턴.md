# 솔루션 아키텍처 패턴

> 사례 중심 설계 연습 — WhatIsTheTime, MyClothes, TODO 앱 등 점진적 아키텍처 진화.

## 개요

새 서비스를 배우는 단원이 아니라, 앞서 본 조각들([[EC2]]·[[ELB & Auto Scaling]]·[[RDS & Aurora & ElastiCache]]·[[Route 53]]·[[EBS & EFS]])을 하나의 웹 서비스로 조립하는 연습이다. 작게 시작한 앱이 트래픽이 늘면서 어떤 문제에 부딪히고, 그때마다 무엇을 더해 푸는지를 단계별로 따라간다. SAA 시험의 시나리오 문제가 정확히 이 사고 흐름("이 요구사항이면 뭘 추가해야 하나")을 묻기 때문에, 사례별로 **문제 → 추가한 조각 → 이유**를 기억하는 게 목표다.

세 가지 사례로 진행한다.

- **WhatIsTheTime.com** — DB가 필요 없는 stateless 앱. 확장(scaling)과 가용성 문제를 푼다.
- **MyClothes.com** — 장바구니가 있는 stateful 앱. 세션 상태를 어디에 둘지 푼다.
- **MyWordPress.com** — 이미지 업로드 + MySQL이 있는 앱. 공유 저장소와 DB 확장을 푼다.

### 사례 1: WhatIsTheTime.com — Stateless 앱의 확장과 가용성

현재 시간만 알려주는 앱. DB가 필요 없고, 처음엔 다운타임을 감수하지만 결국 무중단으로 수직·수평 확장하고 싶다. 진화 순서:

1. **EC2 1대 + Elastic IP** — 가장 단순. 고정 public IP 하나로 접속.
2. **수직 확장(Scale Up)**: 인스턴스 타입을 키움(T2 → M5). **업그레이드 중 다운타임**이 생긴다.
3. **수평 확장(Scale Out)**: 인스턴스를 여러 대로. 이제 IP가 여러 개라 **Route 53**으로 묶는다. 단, 인스턴스에 직접 거는 A 레코드는 TTL 때문에 인스턴스가 사라져도 한동안 옛 IP가 캐싱되는 문제가 있다.
4. **ELB 추가**: 앞단에 로드 밸런서를 두고 **Health Check**로 죽은 인스턴스를 빼낸다. 인스턴스는 private으로 숨기고, 보안 그룹은 **ELB에서 오는 트래픽만** 받게 제한. Route 53은 ELB를 가리키는 **Alias 레코드**로 바꾼다.
5. **Auto Scaling Group(ASG)**: 부하에 따라 인스턴스 수를 자동 조절.
6. **Multi-AZ**: ASG를 여러 AZ에 펼쳐 한 AZ가 죽어도 견디게.
7. **비용 최적화**: 항상 떠 있는 최소 용량(minimum capacity)은 **Reserved Instances**로 잡아 비용을 줄인다.

### 사례 2: MyClothes.com — Stateful 앱의 세션 보관

장바구니가 있고 동시 사용자가 수백 명. 수평 확장은 유지하되 앱은 최대한 stateless로 만들고, **사용자가 장바구니를 잃지 않아야** 한다. 핵심은 "여러 인스턴스에 요청이 분산될 때 세션 상태를 어디에 두느냐"다.

| 방법 | 어떻게 | 한계 / 시험 포인트 |
| --- | --- | --- |
| **ELB Stickiness(세션 고정)** | 같은 사용자를 늘 같은 인스턴스로 보냄 | 그 인스턴스가 죽으면 세션 유실. 부하가 한쪽에 쏠림 |
| **사용자 쿠키에 담기** | 장바구니 내용을 클라이언트 쿠키로 들고 다님 → 앱은 stateless | 요청이 무거워짐, **쿠키 변조 위험(검증 필수)**, **쿠키 4KB 미만** 제한 |
| **서버 세션(ElastiCache)** | 쿠키엔 session_id만, 실제 세션 데이터는 [[RDS & Aurora & ElastiCache\|ElastiCache]]에 저장 | 가장 권장. **DynamoDB도 대안** |

- 사용자 정보(주소·이름 등 영속 데이터)는 **RDS**에 저장한다(세션과 구분).
- 읽기 부하가 크면 **RDS Read Replica**로 읽기를 분산하고, RDS에서 읽은 데이터를 **ElastiCache로 캐싱(Lazy Loading)** 해 DB 부하를 더 던다.
- 가용성: ElastiCache와 RDS 모두 **Multi-AZ**로.
- 보안: 보안 그룹이 **서로를 참조**하게 한다(ELB→EC2, EC2→ElastiCache, EC2→RDS). IP/CIDR이 아니라 보안 그룹 ID로 여는 게 핵심.

### 사례 3: MyWordPress.com — 공유 저장소와 DB 확장

이미지 업로드를 모든 인스턴스가 똑같이 보여줘야 하고, 블로그 글·사용자 데이터는 MySQL에 둔다.

- **DB**: [[RDS & Aurora & ElastiCache\|Aurora MySQL]]로 Multi-AZ + Read Replica를 쉽게 구성.
- **이미지 저장 — EBS vs EFS (시험 단골 비교)**:

| | EBS | EFS |
| --- | --- | --- |
| 성격 | 한 AZ에 묶인 블록 스토리지 | 여러 AZ에서 공유하는 네트워크 파일시스템 |
| 적합 | **단일 인스턴스** 앱 | **여러 인스턴스가 같은 파일 공유**(분산 앱) |
| 다중 AZ | 인스턴스마다 따로(공유 안 됨) | AZ마다 ENI(마운트 타깃)로 같은 파일시스템 공유 |

→ WordPress처럼 여러 인스턴스가 같은 업로드 이미지를 봐야 하면 **EFS**가 답.

### 빠른 인스턴스 기동 (Instantiating quickly)

풀 스택을 새로 띄울 때 설치·설정 시간을 줄이는 방법.

- **EC2**: ① **Golden AMI**(앱·OS 의존성을 미리 설치해 이미지로 구움) ② **User Data 부트스트랩**(부팅 시 동적 설정) ③ **하이브리드**(둘을 섞음, Elastic Beanstalk 방식).
- **RDS**: 스냅샷에서 복원하면 스키마·데이터가 이미 준비된 상태로 뜬다.
- **EBS**: 스냅샷에서 복원하면 포맷·데이터가 된 디스크로 뜬다.

### 전형적인 3계층 구조

- **Public Subnet**: [[Route 53]] + ELB
- **Private Subnet**: ASG로 묶인 EC2 인스턴스(앱 계층)
- **Data Subnet**: ElastiCache(세션·캐시) + RDS(영속 데이터)

### Elastic Beanstalk (여기선 큰 그림만)

위 3계층 구조는 대부분의 웹 앱이 똑같이 반복한다. 개발자는 인프라 관리 말고 **코드만 돌리고 싶다** — 이걸 대신해 주는 관리형 서비스가 Elastic Beanstalk다. EC2·ASG·ELB·RDS를 자동으로 깔아주고, 개발자는 앱 코드만 책임진다. Beanstalk 자체는 무료고 그 아래 뜨는 인스턴스 비용만 낸다. 배포 모드는 **Single Instance(개발용)**와 **High Availability with Load Balancer(운영용)**로 나뉜다. 자세한 구성·배포 정책은 [[Elastic Beanstalk]] 노트에서(3단계 DVA).

## 시험 함정

- 여러 인스턴스가 **같은 파일 공유** 필요 → EBS 아니라 **EFS** #exam/trap/architecture
- 세션 상태를 stateless하게 외부 보관 → **ElastiCache**(또는 DynamoDB), RDS는 영속 사용자 데이터용 #exam/trap/architecture
- 읽기 부하 분산은 **RDS Read Replica**, 가용성(DR)은 **Multi-AZ** — 둘은 목적이 다름 #exam/trap/architecture
- 계층 간 접근 제어는 CIDR이 아니라 **보안 그룹이 서로를 참조** #exam/trap/architecture
- Route 53으로 ELB를 가리킬 땐 **Alias**(루트 도메인 가능, 무료, TTL 자동) #exam/trap/architecture
- 쿠키 보관 방식은 **4KB 미만 + 변조 검증** 제약 #exam/trap/architecture
- 빠른 기동: 앱 미리 구움 → **Golden AMI**, 부팅 시 설정 → **User Data** #exam/trap/architecture
