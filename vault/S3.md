# S3

> 객체 스토리지 — 버킷 정책, 스토리지 클래스, 수명 주기, 복제, 암호화.

## 개요

"무한히 늘어나는" 객체 스토리지. 파일(객체)을 **버킷**이라는 그릇에 담는다. 많은 웹사이트와 AWS 서비스가 S3를 바탕으로 돌아간다. 백업·아카이브·재해 복구·데이터 레이크·정적 웹사이트 호스팅 등에 쓴다.

- **버킷**: 리전 단위로 만든다(글로벌처럼 보여도 실제론 리전에 속함). 이름은 **전 계정·전 리전에서 유일**해야 한다. 제약: 대문자·언더스코어 금지, IP 형식 금지, 소문자나 숫자로 시작, `xn--`로 시작 금지, `-s3alias`로 끝나기 금지.
- **객체**: 키(key)로 식별한다. **키는 전체 경로**이고 prefix + 객체 이름으로 이뤄진다(예: `my_folder/file.txt`에서 `my_folder/`가 prefix). 버킷 안에 진짜 "폴더"는 없고, 슬래시가 들어간 긴 키 이름일 뿐이다.
- 단일 업로드 최대 5GB, 그보다 크면 **멀티파트 업로드** 필수. 객체 하나 최대 크기는 5TB.
- 객체에는 메타데이터(키-값), 태그(최대 10개, 보안·수명 주기에 활용), 버전 ID(버저닝 켜진 경우)가 붙는다.

### 보안 모델 — 접근을 어떻게 허용하나

S3 접근 제어는 두 갈래다.

- **사용자 기반(User-Based)**: [[IAM]] 정책으로 "이 사용자가 어떤 API를 호출할 수 있나"를 정함.
- **리소스 기반(Resource-Based)**: **버킷 정책**(JSON, 버킷 전체 규칙, 크로스 계정 가능), 객체 ACL·버킷 ACL(더 세밀하지만 요즘은 잘 안 씀, 끌 수 있음).

판정 규칙(시험 포인트): **IAM 권한이 허용하거나(OR) 리소스 정책이 허용하고, 명시적 Deny가 없으면** 접근 가능.

- **버킷 정책 용도**: 버킷 공개, 업로드 시 암호화 강제, 다른 계정에 접근 허용.
- **EC2가 S3에 접근**할 땐 액세스 키를 박지 말고 **IAM Role**을 쓴다.
- **Block Public Access**: 회사 데이터 유출을 막으려는 안전장치. 버킷을 공개할 일이 없으면 켜둔다. **계정 단위로도** 걸 수 있다.

### 정적 웹사이트 호스팅

S3로 정적 사이트를 인터넷에 띄울 수 있다. URL은 `http://버킷이름.s3-website-리전.amazonaws.com` 형태. **403 Forbidden이 뜨면 버킷 정책에서 public read를 허용했는지 확인**한다.

### 버저닝

- **버킷 단위**로 켠다. 같은 키로 덮어쓰면 버전이 1, 2, 3…으로 쌓인다. 실수로 지운 것 복구·롤백에 좋다.
- 버저닝 켜기 전부터 있던 파일은 버전이 `null`.
- 버저닝을 **중단(suspend)해도 기존 버전은 안 지워진다**.

### 복제 (CRR & SRR)

- **원본·대상 버킷 모두 버저닝 필수.** 복제는 **비동기**, S3에 적절한 IAM 권한을 줘야 함. 버킷이 서로 다른 계정이어도 됨.
- **CRR(Cross-Region)**: 규정 준수, 지연 줄이기, 계정 간 복제.
- **SRR(Same-Region)**: 로그 수집, 운영↔테스트 계정 간 실시간 복제.
- 복제를 켜면 **그 이후 새 객체만** 복제된다. 기존 객체는 **S3 Batch Replication**으로 따로 복제.
- 삭제: delete marker는 복제 설정으로 선택 가능. **버전 ID를 찍은 삭제는 복제 안 됨**(악의적 삭제 방지).
- **복제는 연쇄(chaining)되지 않는다.** 1→2, 2→3이 있어도 1의 객체가 3까지 가지 않는다.

### 스토리지 클래스 — 시험 최다 빈출

내구성(durability)은 **모든 클래스 11 9's(99.999999999%)로 동일**하다. 갈리는 건 가용성·AZ 수·최소 보관 기간·검색 속도·비용.

| 클래스 | 언제 | 특징 |
| --- | --- | --- |
| **Standard** | 자주 접근 | 가용성 99.99%, AZ ≥3 |
| **Standard-IA** | 가끔 접근, 빠른 접근 필요 (백업·DR) | 더 쌈, 검색 비용 발생, 최소 30일 |
| **One Zone-IA** | 재생성 가능한 2차 백업 | **단일 AZ — 그 AZ 날아가면 데이터 손실**, 최소 30일 |
| **Glacier Instant Retrieval** | 분기 1회 접근, 밀리초 검색 | 최소 90일 |
| **Glacier Flexible Retrieval** | 아카이브 | 검색 Expedited(1~5분)·Standard(3~5시간)·Bulk(5~12시간), 최소 90일 |
| **Glacier Deep Archive** | 장기 보관 | 검색 Standard(12시간)·Bulk(48시간), 최소 180일 |
| **Intelligent-Tiering** | 접근 패턴 모를 때 | 사용량 보고 자동 이동, **검색 비용 없음**, 모니터링 요금 |

### 수명 주기 규칙 (Lifecycle)

객체를 클래스 간 자동 이동·삭제하는 규칙. prefix나 태그로 대상 좁힘.

- **Transition action**: 다른 클래스로 이동(예: 60일 뒤 Standard-IA, 6개월 뒤 Glacier).
- **Expiration action**: 일정 기간 뒤 삭제(로그 365일 뒤 삭제, **오래된 버전 삭제**, 완료 안 된 멀티파트 업로드 정리).
- 시험 시나리오 예: 썸네일은 재생성 가능 → **One-Zone IA + 60일 뒤 expire**, 원본은 Standard → 60일 뒤 Glacier. 삭제 객체를 30일은 즉시·이후 365일은 48시간 내 복구 → **버저닝 + noncurrent 버전을 Standard-IA → Glacier Deep Archive로 transition**.
- **Storage Class Analysis**: 어떤 객체를 언제 옮길지 추천(Standard·Standard-IA만, Glacier·One-Zone엔 안 됨). 수명 주기 규칙 짤 때 출발점.

### 암호화 — 4가지 방식 구분 (시험 핵심)

| 방식 | 키 관리 | 특징 |
| --- | --- | --- |
| **SSE-S3** | AWS가 소유·관리 | **기본값**(새 버킷·객체 자동 적용), AES-256, 헤더 `"x-amz-server-side-encryption":"AES256"` |
| **SSE-KMS** | [[KMS & 암호화\|KMS]]가 관리 | 사용자 제어 + **CloudTrail로 키 사용 감사**. 업로드 시 GenerateDataKey, 다운로드 시 Decrypt API 호출 → **KMS 할당량(quota) 제약**에 걸릴 수 있음 |
| **SSE-C** | 고객이 AWS 밖에서 관리 | S3는 키를 저장 안 함. **HTTPS 필수**, 요청마다 헤더로 키 전달 |
| **Client-Side** | 고객이 전부 | 보내기 전 직접 암호화, 받을 때 직접 복호화 |

- **전송 중 암호화(SSL/TLS)**: HTTP·HTTPS 두 엔드포인트가 있고 HTTPS 권장. **SSE-C는 HTTPS 필수**. 버킷 정책에서 `aws:SecureTransport: false`를 Deny하면 HTTP 차단.
- 기본 암호화 vs 버킷 정책: **버킷 정책이 먼저 평가**된다. 버킷 정책으로 암호화 헤더 없는 PutObject를 거부해 강제할 수 있음.

### 보안 기타 (SAA 빈출)

- **MFA Delete**: 객체 버전 영구 삭제·버저닝 중단에 MFA 요구. **버저닝 켜져 있어야** 하고, **루트 계정만** 켜고 끌 수 있음.
- **Pre-Signed URL**: 권한 가진 사람이 한시적 URL을 발급(콘솔 1~720분, CLI 기본 3600초·최대 168시간). 받은 사람은 **발급자의 권한을 물려받아** GET/PUT. 프리미엄 콘텐츠 다운로드, 임시 업로드 허용에 씀.
- **Glacier Vault Lock / S3 Object Lock**: **WORM(한 번 쓰면 못 고침)** 모델, 규정 준수·보관용. Object Lock은 버저닝 필수, 모드가 둘 — **Compliance(루트도 못 지움)**, **Governance(특정 권한자만 변경)**. Retention Period(기간 보호) + Legal Hold(기간과 무관하게 무기한 보호).
- **CORS**: 다른 오리진(프로토콜+호스트+포트)에서 버킷에 요청할 때 필요. **시험 단골** — 특정 오리진 또는 `*` 허용.
- **S3 Access Logs**: 감사용으로 모든 요청을 다른 버킷에 기록. **로그 버킷은 같은 리전**. **감시 대상 버킷을 로그 버킷으로 지정하면 안 됨**(로그가 무한히 불어남).

### 성능 / 대용량 (고성능 도메인)

- **기본 성능**: prefix당 초당 3,500 PUT/COPY/POST/DELETE, 5,500 GET/HEAD. prefix 수 제한 없음 → prefix를 나눠 분산하면 더 높은 처리량.
- **멀티파트 업로드**: 100MB 이상 권장, 5GB 이상 필수. 병렬 업로드로 속도 ↑.
- **Transfer Acceleration**: 가까운 엣지 로케이션으로 보내 AWS 내부망으로 대상 리전까지 전달. 멀리 있는 버킷에 빠르게 업로드. 멀티파트와 함께 쓸 수 있음.
- **Byte-Range Fetch**: 특정 바이트 범위만 병렬로 GET. 다운로드 가속, 파일 앞부분(헤더)만 가져오기.
- **S3 Batch Operations**: 기존 객체 일괄 처리(메타데이터·ACL 수정, 복사, **미암호화 객체 암호화**, Glacier 복원, Lambda 호출). 대상 목록은 S3 Inventory + Athena로 뽑음.

## 시험 함정

- 내구성은 **모든 클래스 11 9's 동일**, 갈리는 건 가용성·AZ 수·검색 속도·비용 #exam/trap/s3
- **One Zone-IA는 단일 AZ** — AZ 손실 시 데이터 사라짐, 중요 데이터엔 부적합 #exam/trap/s3
- 복제(CRR/SRR)는 **원본·대상 모두 버저닝 필수**, 켠 이후 새 객체만(기존은 Batch Replication) #exam/trap/s3
- 복제는 **연쇄 안 됨**(1→2→3에서 1의 객체가 3까지 안 감) #exam/trap/s3
- 키 사용을 **CloudTrail로 감사**해야 하면 SSE-KMS, 키를 직접 들고 있으면 SSE-C(HTTPS 필수) #exam/trap/s3
- SSE-KMS는 **KMS 할당량**에 걸려 throttle 날 수 있음 #exam/trap/s3
- MFA Delete·Object Lock은 **버저닝 전제**, MFA Delete는 **루트만** 설정 #exam/trap/s3
- 정적 웹사이트 403 Forbidden → **버킷 정책 public read** 확인 #exam/trap/s3
- 다른 오리진에서 호출 실패 → **CORS** 설정 #exam/trap/s3
- 멀리 있는 버킷에 빠른 업로드 → **Transfer Acceleration**, 대용량 병렬 → 멀티파트 #exam/trap/s3
- 처리량 부족 → **prefix를 나눠** 분산(prefix당 3,500 쓰기 / 5,500 읽기) #exam/trap/s3
