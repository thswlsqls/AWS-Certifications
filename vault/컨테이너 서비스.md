# 컨테이너 서비스

> 컨테이너 실행 옵션 — ECS 시작 유형, 태스크 역할, ECR, EKS, App Runner.

## 개요

Docker는 앱을 컨테이너로 묶어 어디서 돌리든 똑같이 실행되게 한다. VM과 달리 호스트 OS의 자원을 공유해서 한 서버에 컨테이너를 훨씬 많이 띄울 수 있다. 이미지는 저장소(Docker Hub = 공개, **Amazon ECR** = AWS의 사설/공개 저장소)에 보관한다.

AWS에서 컨테이너를 돌리는 길은 크게 둘이다.

- **ECS** — AWS가 만든 자체 컨테이너 오케스트레이터.
- **EKS** — AWS가 운영해 주는 관리형 Kubernetes. 쿠버네티스를 이미 쓰는 조직이 옮겨올 때 고른다.

둘 다 컨테이너를 돌릴 바닥(인프라)을 **EC2(내가 관리)** 또는 **Fargate(서버리스, AWS가 관리)** 중에 고른다. 여기에 더 단순한 배포 수단인 **App Runner**, 마이그레이션 도구 **App2Container**가 붙는다.

### ECS — 시작 유형 두 가지

컨테이너를 띄운다는 건 **ECS 클러스터 위에 ECS 태스크를 띄우는 것**이다. 바닥 인프라를 누가 관리하느냐로 시작 유형이 갈린다.

| | EC2 Launch Type | Fargate Launch Type |
| --- | --- | --- |
| 인프라 | **내가 EC2 인스턴스를 직접 띄우고 관리** | **인스턴스 없음 — 서버리스** |
| 각 인스턴스 | **ECS Agent**를 돌려 클러스터에 등록 | 신경 쓸 것 없음 |
| 내가 하는 일 | 인스턴스 프로비저닝 + 태스크 정의 | **태스크 정의만**, 필요한 CPU/RAM 지정 |
| 확장 | 인스턴스 수도 같이 늘려야 함 | **태스크 수만 늘리면 끝** |

→ 인프라 관리를 하기 싫으면 Fargate. 시험에서 "서버를 관리하고 싶지 않다 / 서버리스 컨테이너"가 나오면 Fargate.

### ECS의 IAM 역할 — 두 개를 구분 (단골 함정)

- **EC2 Instance Profile** (EC2 시작 유형에서만): **ECS Agent**가 쓴다. ECS API 호출, ECR에서 이미지 pull, CloudWatch Logs로 로그 전송, Secrets Manager/SSM Parameter Store의 비밀값 참조에 필요.
- **ECS Task Role**: **태스크(컨테이너)별로** 다른 권한을 준다. 예를 들어 Task A는 S3, Task B는 DynamoDB. **task definition에서 지정한다.**

### ECS — 로드 밸런서·스토리지·확장

- **로드 밸런서**: 보통 **ALB**면 충분. **NLB**는 고성능·초고처리량이거나 AWS PrivateLink와 함께 쓸 때. CLB는 지원하지만 권장 안 함(Fargate 불가).
- **데이터 볼륨**: 태스크에 **EFS**를 마운트하면 EC2·Fargate 양쪽에서, 그리고 여러 AZ의 태스크가 같은 파일을 공유한다(영속 + Multi-AZ 공유 스토리지). **Fargate + EFS = 완전 서버리스**. **S3는 파일 시스템으로 마운트 못 한다.**
- **ECS Service Auto Scaling**(태스크 수 조절, AWS Application Auto Scaling 사용): CPU/메모리 사용률이나 ALB의 요청 수 기준. 방식은 Target Tracking / Step Scaling / Scheduled Scaling.
- **이건 태스크 수준 확장이고, EC2 인스턴스 수 확장과 다르다.** EC2 시작 유형이면 인스턴스도 따로 늘려야 하는데, ASG로 CPU 기준 확장하거나 **ECS Cluster Capacity Provider**로 용량이 모자랄 때 자동으로 인스턴스를 붙인다. Fargate는 인스턴스가 없어 확장 설정이 훨씬 간단.

### ECS와 다른 서비스 연동

- **EventBridge로 태스크 실행**: S3 업로드 이벤트 → EventBridge 규칙 → ECS 태스크를 새로 띄워 처리(Task Role로 S3·DynamoDB 접근). 일정 기반(예: 1시간마다)으로 배치 태스크를 돌릴 수도 있다.
- **SQS와 연동**: 큐를 폴링하는 태스크를 두고 ECS Service Auto Scaling으로 처리량을 늘린다.
- **중단된 태스크 감지**: EventBridge로 `ECS Task State Change`(lastStatus=STOPPED) 이벤트를 잡아 SNS로 관리자에게 알림.

### EKS — 관리형 Kubernetes

- ECS의 대안. 목적은 비슷하지만 **API가 Kubernetes**다. **이미 쿠버네티스를 쓰는** 온프레미스·타 클라우드에서 옮겨올 때 고른다. 쿠버네티스는 **클라우드 중립적**이라 Azure·GCP에서도 같은 방식으로 쓸 수 있는 게 장점.
- 노드(워커) 유형: **Managed Node Groups**(EKS가 EC2 노드를 ASG로 생성·관리), **Self-Managed Nodes**(내가 만들어 등록, EKS Optimized AMI 사용 가능), **Fargate**(노드 관리 없음). Node Group들은 On-Demand·Spot 모두 지원.
- 데이터 볼륨: **StorageClass** 매니페스트 + **CSI 드라이버**로 EBS·EFS(Fargate 가능)·FSx for Lustre·FSx for NetApp ONTAP 연결.
- 여러 리전이면 **리전마다 EKS 클러스터 하나**. 로그·지표는 **CloudWatch Container Insights**.

### App Runner / App2Container

- **App Runner**: 소스 코드나 컨테이너 이미지만 주면 빌드·배포·로드 밸런서·확장·암호화까지 알아서 해 주는 완전 관리형. 인프라 지식 없이 웹 앱·API를 빠르게 운영할 때.
- **App2Container (A2C)**: 온프레미스나 타 클라우드의 **Java·.NET 웹 앱을 코드 변경 없이** 컨테이너로 옮기는 CLI 도구(lift-and-shift). CloudFormation 템플릿을 만들고 ECR에 등록한 뒤 ECS·EKS·App Runner에 배포.

## 시험 함정

- "서버 관리 없이 컨테이너 실행" → **Fargate**. 반대로 EC2 시작 유형은 인스턴스를 직접 띄우고 ECS Agent를 돌려야 함 #exam/trap/containers
- **EC2 Instance Profile**(ECS Agent용, EC2 시작 유형만)과 **ECS Task Role**(태스크별 권한, task definition에 지정)을 헷갈리지 말 것 #exam/trap/containers
- 태스크가 ECR 이미지 pull·CloudWatch Logs 전송에서 권한 오류 → IAM 역할(Instance Profile) 정책 문제 #exam/trap/containers
- "여러 컨테이너/태스크가 같은 파일을 공유" → **EFS** 마운트(EC2·Fargate 모두). **S3는 파일 시스템 마운트 불가** #exam/trap/containers
- **ECS Service Auto Scaling(태스크 수)** ≠ **EC2 Auto Scaling(인스턴스 수)**. EC2 시작 유형은 둘 다 챙겨야 하고, 인스턴스 자동 확장은 Capacity Provider #exam/trap/containers
- "이미 Kubernetes를 쓰는데 AWS로 마이그레이션" → **EKS**. 클라우드 중립이 필요할 때도 EKS #exam/trap/containers
- "기존 Java/.NET 앱을 코드 변경 없이 컨테이너화해 이전" → **App2Container** #exam/trap/migration
